# Agent Operating Guide

This repository is the design and future implementation workspace for **Trading Journal (Option B)** — a purpose-built app for capturing annotated trading-chart reviews and classifying them with a flexible, many-to-many tag model. It replaces a PowerPoint-based journaling workflow where the same annotated chart had to be physically copied into every category it belonged to.

## Source of Truth

- Treat `trading-journal-brief-ch.md` as the current product boundary, domain model, and thesis.
- The user's existing PowerPoint journal is design *input only*. Do not assume the app must replicate PowerPoint's full drawing surface. Replicate the workflow the user actually performs: paste a chart screenshot, annotate it, tag any annotation (or the whole chart) with values from user-defined groups, and later retrieve by any tag combination.
- Tool comparisons (PowerPoint add-in / Obsidian+Excalidraw / custom web app) are background context. Only what the brief states is settled.

## How to Work in This Repo

- Treat product / model / UX design discussion as design work, **not** as permission to change implementation. During design discussion, clarify the domain model, tradeoffs, terminology, and target-state docs first. Do not create app source, build config, or tests until the user explicitly asks to implement, or the design has clearly converged into an implementation slice.
- For domain decisions (Entry, Annotation, groups/tags, view rendering), use the `trading-journal-domain` skill.
- For tag and group naming conventions, use the `tagging-conventions` skill. Which groups exist and what values each holds is user-defined domain vocabulary, not part of any skill.
- For product architecture or MVP scope review, use the `product-design-review` skill.
- For future implementation work, use the `implementation-planning` skill to split changes into small, reviewable slices. The current slice plan lives in `specs/implementation-plan-ch.md`.
- Record implementation status as an **实现状态** note inside each slice's section in `specs/implementation-plan-ch.md`, not here — what each slice/iteration builds; the toolchain, verified commands, and tech decisions live in the Slice 0 note. Keep `AGENTS.md` limited to general, durable guidance (source-of-truth, how-to-work, product invariants, code guidelines); do not let iteration detail accumulate in it.
- Write specs as finished target-state documents. When discussion changes the design, rewrite the affected sections into their final form instead of appending revision notes, addenda, or "based on the latest discussion" paragraphs.
- Prefer precise edits to the smallest coherent set of files. If a design change affects multiple docs, update all of them or explicitly say why one is intentionally left alone.

## Product Invariants

- **Three separate things.** The annotated review artifact, its classification tags, and the source chart are distinct. Never conflate "which category a review belongs to" with "where the drawing physically lives."
- **Entry is the durable artifact.** An Entry = a review **page** (a white canvas of a stored size) + one or more **screenshots** (as movable/stackable image objects) + annotations + entry-level tags. It is stored exactly once and never duplicated to appear in a second category. The list thumbnail is a **rendered snapshot of the page** (`entries.thumbnail`, refreshed on every auto-save, so it reflects all edits); `Entry.image` is now just an optional **cover hash** kept as a pre-render fallback. The page composition (screenshots + annotations) lives in `canvas_json`.
- **No special annotation type.** Any annotation (box, text box, arrow, …) can carry tags from any group. Tagging an entry-point box with a setup tag is just one case; tagging a text box with an intraday-structure tag is another — same mechanism. "The tagged element is briefly highlighted when you browse that tag" is general behavior; setup is not special.
- **Notes are taggable text-box annotations.** A note is a text-box annotation whose text is the note and which can itself carry tags. There is no separate note field.
- **Internal references are stable targets plus text hyperlinks.** Every journal has a persistent immutable `journalId`; every Entry and annotation has an immutable-id address `trading-journal://journal/{journalId}/{entry|annotation}/{targetId}` with each id encoded as one canonical path segment. Right-click copies that address without mutating the journal. Editable title/text-box character ranges may carry typed `textLinks` to same-journal targets; Fabric JSON stores them as `tjTextLinks`, while public domain/API fields stay `textLinks`. Display text is the underlying text range, not duplicated link data. There is no annotation-to-annotation `links[]`, app-only link clipboard, or object-level Links popover.
- **Classification is grouped tags, not folders.** A "section" is a saved query over tags, not a physical container. A tag is `group:value`; tags attach at Entry level or annotation level; annotation-level tags decide which element is highlighted; `date` is the one structural, always-present group.
- **Outcome is a typed annotation result, not a tag.** A trade's outcome is an optional, multi-dimensional, typed `result` on an annotation (each dimension a `string` or a `number`; dimensions are user-defined). It may be used as a **filter predicate in a view's annotation dimension** (string equality/set, number range/threshold) and is aggregated in statistics, but it is never a `group:value` tag, never a pivot/browse bucket dimension, and never drives the highlight transform; it must not be modeled as an `outcome` group/tag.
- **User owns the vocabulary and reusable components.** Which groups exist, their values, and which reusable stamps exist are all user-defined/user-designed. The app ships the mechanism (grouped tags, drawing primitives, a stamp library), never a fixed default catalog. Names in these docs are illustrative examples, not built-in defaults.
- **One Entry, many views, zero copies.** The same Entry renders under any group/tag browse without a second stored copy. Daily Review is just browsing the `date` group.
- **Highlight is a non-destructive render transform.** Browsing a tag briefly highlights the annotations carrying it (no viewport fit, no persistent dimming) — computed at render time from the active tag and annotation bounds, never persisted.
- **V1 annotates static chart screenshots on a page.** The white page is the review surface; screenshots are image objects on it (paste/stack/resize/reorder), referenced in `canvas_json` by `tj-image://<hash>` (bytes live in `images/<hash>`, never base64 in JSON). Annotation and object geometry is stored in page-pixel coordinates. Do not bind annotations to price/time axes until a live/replayable chart source is an explicit product goal.
- **Scope guard.** Do not turn the app into a charting platform, a broker/execution tool, a live-market-data feed, a backtesting engine, or a general-purpose drawing tool.
- **AI Access is optional and permanently read-only.** Start is the single explicit grant for compatible local agents to read the current journal's structured queries, statistics, machine-readable text, and visual evidence; Stop revokes that access. Do not build per-agent or per-data permissions. MCP must never expose create/update/delete/tag/result/note/save, SQL, filesystem paths, filesystem writes, or AI-output persistence. Image “export” means returning revision-bound bytes/resources, checksums, and suggested filenames; an external coding agent may use its own independent filesystem tools to save those bytes into the user's repo or research directory. Trading Journal never knows or writes that destination. Trading Journal does not embed a model or agent; external agents are untrusted consumers.

## Future Code Guidelines

- Do not start implementation from an unresolved design conversation. Confirm the design decision or implementation slice first, then edit code.
- Define stable data-model contracts before UI: `Entry`, `Annotation`, `Tag` (`group:value`), the annotation-level typed `Result` (string/number dimensions, stats-only), `SavedView`/query. Keep them typed and schema-checked, not ad hoc strings.
- Before placing any module, decide its ownership boundary first: **Ingest** (screenshot import), **Entry store** (durable persistence, internal-link target resolution, and text-link projection), **Annotation-Tag index** (denormalized annotations + tags + typed results for querying), **Tag & query engine** (group/tag parsing, boolean tag queries, counts, result statistics), **Canvas/annotation layer** (drawing + shapes with attached tags/results and editable-text `textLinks`), or **Render/view layer** (browse by group/tag; brief-highlight transform). A module that reads/writes Entry, Annotation, Tag, Result, SavedView, or TextLink state belongs to one of these boundaries, not scattered across the UI.
- Never satisfy a view by duplicating an artifact. A view is a query plus a render mode.
- Keep highlight/browse state out of persistence. It is derived from the active tag and the annotation bounds.
- Prefer structured payloads, typed groups, and schemas over free-form tag strings. A tag is `group:value` with a stable, kebab-cased value.
- Any panel, dialog, or list that can grow with user data (Settings groups/values, Result types, saved views, browse buckets, …) must cap at a sensible length and scroll its long region — never grow unbounded with content or screen size. Bound the container height (e.g. `max-height: min(vh, px)`) and put the growing part in a scrollable child (`min-height:0; overflow-y:auto`); the shared `.modal__panel` already encodes this for dialogs.
- Avoid hidden special cases that encode a desired net effect. A category is not a hardcoded branch; it is a tag value matched by the query engine.
- AI Access implementation must use a dedicated closed `JournalReadApi`, a readonly SQLite connection with `PRAGMA query_only=ON`, and a supervised companion that receives neither the DB path nor mixed `IpcApi`. Do not create a generic invoke method, arbitrary SQL/file resource, filesystem write/export API, future journal-write hook, AI-specific copy of query/stats logic, persistent report, or second artifact store. Keep the Supervisor / read repository / Fabric / Sharp stack behind the explicit Start path; AI Access Off must not load those heavy modules during normal app startup. The user-editable Agent Guide / Prompt Library is machine-local app configuration and MCP exposes it read-only via resources/prompts; agents cannot edit it. Visual grounding and byte export must share one single-Entry, revision-bound artifact pipeline with exact Fabric object transforms, screenshot-instance ownership, deterministic annotation marks, and paired marked/clean native crops where safe; an axis-aligned annotation bound alone is not sufficient. A bounded chunk reader may expose an artifact's bytes plus checksum and suggested filename, but it never accepts a destination path; saving into a repo is the external agent's own action. Progressive bar reveal is a source-pixel mask over a user/agent-calibrated uniform spacing, not chart replay, candle detection, or structured price/time/bar facts. Prove journal non-mutation with a canonical domain digest and keep the golden-DB test green.
- AI research should optimize for a few information-dense calls: resolve IDs/levels with vocabulary usage counts, call `prepare_sample_study` for exact population/denominators/context/visual planning, then call `get_visual_evidence_batch`. Keep low-level search/context/single-visual tools for drill-down, not as the default research loop. Tool failures must expose stable, actionable error codes and hints while still withholding paths, SQL, and stacks.
- **From v0.1.0 the persisted journal is a preserved data contract — never break it.** The data folder holds real user reviews (`app.sqlite`, `images/<hash>`, each entry's `canvas_json`) plus the separate machine-local `config.json` pointer. Every bug fix or feature must keep existing journals openable and their data intact, or ship an explicit forward migration that transforms them with zero data loss. There is no more "just delete the dev data" for user journals (throwaway test temp dirs are still disposable):
  - **SQLite schema is append-only.** Evolve it only by appending a new function to the `MIGRATIONS` array in `db.ts` (which bumps `user_version`); never edit, reorder, or delete an existing migration, and each migration must preserve or transform existing rows, never drop user data.
  - **Two runtime guards stay.** `openDatabase` snapshots `app.sqlite` into `backups/` before applying any pending migration, and throws `JournalTooNewError` (refusing to open) when a journal's `user_version` exceeds the app's schema — an older app must never write a newer journal.
  - **`canvas_json` is part of the contract.** Its embedded annotation/text shape (`tjId`, tags, result, typed `tjTextLinks` corresponding to public `textLinks`, page geometry, `tj-image://<hash>` refs) must not change format silently; a shape change needs a load-time upgrade path, never a break.
  - **Prove it with the golden-DB test.** `tests/e2e/data-migration.spec.ts` opens the committed `tests/fixtures/journal-v*.sqlite` and asserts every review survives the migration chain — keep it green, and when a migration transforms data, extend it; snapshot a fresh fixture at the new schema when a release lands.
  - Bump `package.json` `version` for releases; a schema or data-contract change implies a migration plus a test. Only outside this data contract (pure UI/logic with no persisted footprint) may you still change code outright without a compatibility path.
- Fix at the root cause, never a symptom or an opportunistic workaround. When something misbehaves, debug in this order — is the domain model wrong, is the implementation violating the model, or is an explicit domain validation missing — and fix that cause. Do not special-case to force a green result, loosen or delete a test to make it pass, hardcode an expected value, or add sleeps/retries to hide a race; if the proper fix is large or unclear, surface it instead of hacking around it.
- Tests must assert domain behavior and public contracts: an annotation tagged `group:<value>` appears in that tag query; an Entry appears under any browsed tag without a second stored row; highlight geometry is derived, not stored. Do not assert source-code shape.
